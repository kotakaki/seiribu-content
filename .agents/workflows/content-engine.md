# SEO記事制作エンジン (v2.0) for セイリ部

キーワードとトピックを入力として、構成承認を経た高品質な記事を段階的に作成します。
本ワークフローは `seiribu-editorial/strategy/` 配下の戦略ドキュメントに準拠します。

> [!IMPORTANT]
> **【AIへの重要指示：出力フォーマットの事前ロード】**
> 本ワークフローを起動されたAIは、**PHASE 1 に入る前に** `view_file` で `.agents/workflows/article-output-format.md` を全文読み込むこと。
> 凍結済み設計図の構造、導入文の6要素構成（①背景提示（問いかけ禁止・一般論） → ②読者の心の声（鉤括弧） → ③あるあるの言語化と客観的分析 → ④課題と解決策の提示と宣言 → ⑤Kacky吹き出し → ⑥「この記事でわかること」箇条書き）、AI特有のツギハギ・冗長表現の禁止、画像の制作ブリーフ形式、FAQ形式、frontmatter禁止など、すべての出力ルールはこのファイルに定義されている。
> **PHASE 0〜3 のすべてのフェーズで `article-output-format.md` のルールに厳密に従うこと。**

## 起動方法
`/content-engine [対象キーワード] [トピック/戦略]`

---

# ⛔ 【最重要：ステートマシン制約】

> [!IMPORTANT]
> 1. **フェーズ宣言**: 出力の冒頭で必ず `【現在実行中のフェーズ: PHASE X】` と宣言すること。
> 2. **スコープ制限**: 指定されたフェーズ以外の内容は出力しないこと。
> 3. **強制停止**: 各フェーズの最後は必ずユーザーへの確認で終え、**出力を完全に停止**すること。
> 4. **ファイル出力**: 成果物はチャットではなく、適切なディレクトリ（例：`seiribu-editorial/drafts/` 配下）の `.md` ファイルに保存・追記すること。

# 📝 全体フォーマット・品質定義

## 最上位原則
- **検索意図への直答**: 読者の検索意図にシンプルに答えることが最優先。情報の網羅性より、意図への的中率を重視する。「この情報は読者の疑問に答えているか？」を常に自問し、答えがNoなら書かない。

## SEO/GEO戦略
- **First Answer Rule**: 最初のH2直下で、読者の検索意図に対する回答を即座に提示する。
- **キーワード配置**:
  - タイトルおよびすべてのH2にターゲットキーワードまたは関連キーワードを含める。
  - **導入文の冒頭（背景提示→心の声→あるあるの言語化→解決策宣言）**: まず地の文（問いかけ禁止・客観的事実として）でターゲットKWを含む背景を提示し、鉤括弧で読者の生々しい悩みを代弁する。次に「あるある」を一般現象として言語化して客観的に分析し、課題の提示から解決策と「この記事では〜を解説します」の宣言へ自然に繋ぐ。Kackyの吹き出しで背中を押し、「この記事でわかること」の箇条書きで締める。AI特有のツギハギで冗長な文章を厳しく禁止する。
- **エバーグリーン原則**: タイトルや本文に特定の年号（2026年など）を原則含めない。

## ペルソナ・編集方針
- **ペルソナ**: `AGENTS.md` に定義された「信頼できる案内人」として執筆する。
- **Seiribu Editorial Rules**:
  - **実体験の尊重**: 実体験がないのに捏造して書かない。
  - **判断優先**: アフィリエイト誘導より、読者の「売る・残す・捨てる」の判断を優先する。
  - **禁止表現**: 「第〇章」「前章では」等の書籍的表現、および水平線（`---`）の挿入を禁止する。

## 太字（Bold）フォーマットルール
- **括弧と太字の位置関係**: 鉤括弧「」や丸括弧（）を太字にする場合、`**` は必ず括弧の**内側**に置く。正：「**テキスト**」、誤：**「テキスト」**
- **強調は最小限に**: 本当に重要な箇所のみ太字にし、乱用しない。アトミック・アンサー（結論、150文字以内）では重要な箇所のみを太字で強調する。
- **自然な文脈に溶け込ませる**: 不自然な鉤括弧囲みは避け、文章の流れの中で太字を使う。

## PHASE 0：リサーチとコンテキスト統合
1. **検索意図の分析**: `search_web` でターゲットKWを検索し、読者が本当に知りたいことを特定する。
2. **SERP確認（必須）**: `search_web` の要約に頼らず、上位5件の記事の実際のH2構造（見出し）とスニペットを取得すること。必要に応じてPythonスクリプト等を書き、生の検索結果データを分析し、以下を把握する。
   - 上位記事が共通して扱っているトピック（＝読者が期待する最低限の内容）
   - 上位記事が扱っていない、またはセイリ部だからこそ書ける独自の切り口
   - ※目的は「競合と同じ記事を書くこと」ではなく、読者の意図に対する差分を見つけること。
3. **戦略合致確認**: 戦略ドキュメントに照らし、この記事がどの階層（ハブ/買取判断/家族心理等）に属するかを特定する。

**出力:** 検索意図の分析結果、SERP上位との差分、戦略上の位置づけ、構成案作成への移行確認。

## PHASE 1：構成案の設計
記事の「設計図」を作成し、チャット画面を汚さずに直接ファイルへ書き込む。

**アクション指示:**
1. 以下を含む構成案を設計する。
   - ターゲット読者と検索意図
   - 記事タイプ（CV型/教育型/思想型）
   - Out of Scope（扱わないこと）
   - タイトル案（3案検討し、最適な1案を採用する）
   - メタデータ（スラッグ、ディスクリプション）
   - 導入文ドラフト（【重要】背景提示（地の文でKW提示・問いかけ禁止） → 読者の心の声（鉤括弧） → あるあるの言語化と客観的分析 → 課題と解決策の提示と宣言 → Kackyの吹き出し → 「この記事でわかること」4つの箇条書き。※そのままファイルに書き込まれるため完成品質で作成する。詳細は `article-output-format.md` のテンプレートに準拠）
   - 読後判断と10秒判断ブロック設計（記事を読んだ後の行動と、状況別の分岐を設計する）
   - 見出し構成案（H2・H3一覧）
   - FAQ設計案（読者が検索しそうな疑問を3〜5問。設計図の【FAQ設計案】に記載し、PHASE 3で本文に展開する）
   - CTA・導線設計案（Conversion momentや配置場所を定義）
2. 構成案が完成したら、**チャット画面には長文を出力せず**、該当ディレクトリ（例：`seiribu-editorial/drafts/`）に `[slug].md` を**直ちに新規作成（または上書き更新）**する。
3. **ファイルの構造は `article-output-format.md` に100%準拠すること。** 冒頭に構成案を `<!-- === 凍結済み設計図（Invariants） === ... === END === -->` のHTMLコメントとして埋め込み、その直後に **H1（タイトル）、導入文（完成版）、そして「10秒判断ブロック」** を書き込んでファイルを保存する。
4. チャット画面では「`[slug].md` に設計図と導入文を作成しました。エディタ上で内容を確認・修正してください」とだけ短く報告し、承認（PHASE 2への移行許可）を待つ。

**【読者の疑問チェーンとSEO要件の事前確認】**
見出し構成案を作成したら、ファイルへ書き込む前に以下を自己チェックし、一つでもNoがあれば修正すること。
- [ ] **SEO絶対要件**: すべてのH2（特に最初のH2）にターゲットKWまたは関連KWが自然に含まれているか？
- [ ] H2の並び順は「このH2を読んだ読者が次に抱く自然な疑問」の順番になっているか？
- [ ] 各H2の内容は、読者がその時点で知りたいことに過不足なく答えているか（余計な情報で膨らませていないか）？
- [ ] 最後のH2まで読んだ読者が「知りたかったことは全部わかった」と感じられるか？

---

## PHASE 2：セクション別・精密執筆
1つのH2セクションずつ順番に執筆し、ファイルを更新する。

**【重要：強制停止ルール】**
- 1回の出力で書くのは「1つのH2とその配下のH3」のみです。
- 1つのH2セクションを書き終えたら、ユーザーへ確認を求め、**出力を完全に停止**してください。決して複数のH2を一気に（暴走して）書かないこと。
- **最後のH2セクション（まとめの直前）の執筆が完了し承認を得たら、記事執筆を終了せず、直ちに PHASE 3（FAQ・自己検証）の提案を行い、ユーザーの承認を待つこと。**

**セクション構造:**
- リード文
- アトミック・アンサー（結論・太字活用）
- 詳細解説（論理的深掘り）
- 具体的インサイト/実例
- 画像ブリーフ（`article-output-format.md` の `> [!CMS担当者へ：` 形式で、セクション内の適切な箇所に配置する。すべてのH2に必須ではないが、図解や情景イラストが読者の理解を助ける箇所には積極的に配置すること）

**【品質心得】** 導入文だけでなく、すべてのセクションで Seiribu Voice（誠実・知的・共感的・実務的）と品質心得（AI特有のツギハギ感の排除、滑らかな接続、冗長の禁止）を適用すること。

---

## PHASE 3：最終化と最適化
1. FAQ（構造化データ用）
2. まとめ（結論とNext Step）
3. 自己検証チェックリスト（以下をすべて確認し、結果を報告すること）
   - [ ] 導入文の概要宣言（「この記事では〜」）にターゲットKWが自然に含まれているか
   - [ ] 最初のH2見出しにターゲットKWが含まれているか
   - [ ] すべてのH2にターゲットKWまたは関連KWが含まれているか
   - [ ] 太字の `**` が括弧の内側に置かれているか（全箇所）
   - [ ] 本文中に「PHASE」「H2-1」等の制作用ラベルが残っていないか
   - [ ] 本文中に水平線（`---`）が使われていないか
   - [ ] 記事冒頭に固定の挨拶文（こんにちは。〜）が入っていないか（挨拶はトップ・ピラーのみ）
   - [ ] ピラー記事の場合、冒頭に固定の挨拶文が含まれているか
   - [ ] 導入文が「背景提示（問いかけ禁止・客観的描写）」→「読者の心の声（鉤括弧）」→「あるあるの言語化と客観的分析」→「課題と解決策の提示と宣言」→「Kackyの吹き出し」→「箇条書き」の順になっているか
   - [ ] 導入文の直後（最初のH2の前）に「10秒判断ブロック」が正しく配置されているか
   - [ ] 各H2の並び順が「読者の次の疑問」の自然な流れになっているか
   - [ ] FAQセクションのH2見出しにターゲットKWが含まれているか（例：`## 〇〇に関してよくある質問`）
   - [ ] アイキャッチの画像ブリーフ（`> [!CMS担当者へ：アイキャッチ画像]`）が記事末尾に配置されているか
   - [ ] 本文中の画像ブリーフがすべて `> [!CMS担当者へ：` 形式になっているか（旧式の `<!-- === 画像制作ブリーフ ===` 形式が残っていないか）

**完了後のアクション:**
自己検証チェックリストをすべてクリアしたら、ユーザーに対して以下のスラッシュコマンドを順番に実行するよう促し、自身の処理を終了すること。
1. `/image-engine` （アイキャッチと記事内画像の制作プラン生成）
2. `/revenue-engine` （アフィリエイト案件の最適な挿入）
