# セイリ部 画像制作エンジン

完成した記事ドラフトを読み込み、記事内の `> [!CMS担当者へ：...]` ブロックを **CMSブリーフ** として解釈し、Codexへ渡す `image-plan.json` / `image-plan.md` を生成するワークフローです。

この記事制作側の役割は、完成プロンプトを記事本文に埋め込むことではありません。記事本文には、WordPress手動投入時に使う仮メモとして、画像の目的、設置位置、ALT、キャプション候補、入れたい要素、避けたい要素を残します。画像を貼り終えたら、このCMSブリーフは記事本文から削除します。画像生成、ロゴ合成、軽量化、README更新はCodex画像仕上げレーンで行います。

## 基本方針

- **画像枚数の固定:** 各記事には、アイキャッチ用画像素材1枚と、記事内画像または図解4枚の合計5枚を用意する。
- **CMSブリーフはWordPress作業用メモ:** 記事内には、画像の目的、読者に伝えたいこと、入れたい要素、避けたい表現、ALT、キャプション候補、設置位置を書く。これは画像貼り付け後に削除する仮メモであり、公開本文には残さない。
- **完成プロンプトは記事内に書かない:** 英語のChatGPT用プロンプトやDALL-E指定は `image-plan.json` / `image-plan.md` 側に出す。記事本文には残さない。
- **画像プランJSONを機械正本にする:** `scripts/generate_image_plan.py` で `image-briefs/<slug>-image-plan.json` と確認用Markdownを生成する。Codexが読む正本はJSON、人間が見る要約はMarkdown。
- **画像生成はCodex側:** Antigravity側では画像を生成しない。Codexが `image-plan.json` を読み、`seiribu-image-finisher` スキルでアイキャッチ素材と本文画像・図解を生成する。ここで止めず、ロゴ合成、保存、README更新まで進める。
- **アイキャッチ完成版はCanva手動:** Canvaでしか使えないフォントを使うため、Codexはアイキャッチ素材まで自動生成し、タイトルとロゴの最終合成は人間がCanvaで行う。
- **本文画像・図解はCodexで完成:** 記事内画像・図解4枚はCodex側で生成、ロゴ配置、サイズ調整、可能な範囲の軽量化まで行い、WordPressへ手動アップロードできる状態にする。
- **WordPressアップロードは手動:** Codexはアップロード用画像一式まで作る。WordPressへの画像アップロードと公開操作は人間が行う。

## 役割分担

| 役割 | 担当すること | 担当しないこと |
| --- | --- | --- |
| Antigravity記事制作レーン | 記事本文、CMSブリーフ、`image-plan.json` / `image-plan.md` 生成 | 画像生成、ロゴ合成、WordPressアップロード |
| Codex画像仕上げレーン | 画像生成、本文画像・図解のロゴ合成、軽量化、README更新、アップロード用画像一式作成 | 記事本文制作、Canvaでのアイキャッチ最終合成、WordPressアップロード |
| Canva手動仕上げ | アイキャッチ素材にタイトル、サブタイトル、ロゴ、Canva専用フォントを載せる | 本文画像4枚の標準仕上げ |
| 人間 | 最終確認、Canva仕上げ、WordPress手動アップロード、公開判断 | なし |

## 記事内CMSブリーフの書き方

記事内には、以下のWordPress作業用メモを書く。これは画像の挿入位置、ALT、キャプションをコピペするために必要なので残す。完成したChatGPT用プロンプトやDALL-E指定だけは記事本文に書かず、画像プラン側へ分離する。

```markdown
> [!CMS担当者へ：ここに画像を挿入]
> 種類：アイキャッチ素材 / 記事内イメージ / 記事内図解 / 写真風素材
> 目的：この画像で読者に伝えたいこと
> 読者に伝えたい感情：安心感、警戒心、納得感など
> 入れたい要素：人物、場所、品物、場面、必要な短いラベル
> 避けたい表現：暗い遺品整理感、ゴミ屋敷感、高額査定の強調など
> ALT：画像を見られない人にも伝わる説明
> キャプション候補：必要な場合のみ
> 設置位置：該当H2直下、表の前後、まとめ前など
```

## 画像プランの生成

記事が完成したら、以下で画像プランを生成する。

```bash
python scripts/generate_image_plan.py --mode standard drafts/example.md
```

標準では5枚分を保持する。軽量確認だけしたい場合を除き、本文画像4枚を保留にしすぎない。

生成されるファイル:

- `image-briefs/<slug>-image-plan.json`: Codexが読む機械正本
- `image-briefs/<slug>-image-plan.md`: 人間が見る確認用要約

## Codexへの投入

画像プラン生成後、CodexにはJSONパスをそのまま渡す。

```text
/Users/kota/Documents/アンチグラビティ/seiribu-content/seiribu-editorial/image-briefs/<slug>-image-plan.json
このJSONをもとに画像を生成し、ロゴ合成、保存、README更新まで完了してください。
```

Codex側では `seiribu-image-finisher` スキルを使い、画像生成で止めずに `assets/images/<slug>/` まで成果物を出す。

## 画像タイプ別ルール

| 画像タイプ | Antigravity側で書くこと | Codex側で行うこと |
| --- | --- | --- |
| アイキャッチ素材 | 記事の第一印象、入れたい人物・物、避けたい表現、Canvaで載せるタイトル/サブタイトル | 文字・ロゴなしの素材を生成。Canvaで仕上げやすい余白を確保 |
| 記事内イメージ | H2の文脈、読者の感情、人物、場所、品物、避けたい表現 | サンプルのような温かい実家整理イラストを生成し、ロゴを合成 |
| 記事内図解 | 伝えたい判断、手順、比較、必要な短いラベル、使いたい場面 | 抽象的な線や図形だけにせず、人物・品物・部屋のあるイラスト図解として生成または合成 |
| 写真風素材 | 実物感が必要な理由、写す品物、避けたい広告感 | 必要な場合だけ写真風生成。暗さや業者広告感を避ける |

## 図解品質の絶対ルール

前回のような、線・矢印・丸・四角だけを組み合わせた低品質な図形画像は不可。

セイリ部の図解は、**スッキリとした背景と余白を活かした構造的なインフォグラフィック（図表）**であることを基本とします。

- 画面いっぱいに広がる漫画のようなコマ割り（全画面情景パネル）は避ける。
- 背景は白や薄いベージュなどでスッキリさせ、各ステップや比較を「枠線つきのボックス」に収めるなど、構造的に見せる。
- 人物のイラストや品物（段ボール、着物、本、カメラなど）を含めるのは良いが、あくまで図解の一部（カットイラストやアイコン的扱い）としてシンプルに配置する。
- 意味を分かりやすくするため、**図解内の文字（テキストラベルや短い説明文）は積極的に含めて良い。**
- 画像単体でPinterestやGoogle画像検索に出ても意味が伝わる構成にする。
- ロゴはCodex側で正確に合成する。

良い図解の方向性:

- 枠線ボックスと矢印を使った構造的なステップ図表（人物のカットイラストを含む）
- スッキリした背景に品物と金額アイコンを並べた計算・判断図解
- OK/NGをシンプルに対比させた図解
- 品物を並べた判断イラスト

避ける図解:

- 抽象的な線、丸、四角、矢印だけの図
- 意味の薄いフローチャート
- 読めない日本語が入った画像
- アイコンだけで生活感がない図
- 無料素材風の薄い説明画像

## ブランド帰属ロゴ

本文画像・図解4枚にも、ブランド帰属ロゴを必ず入れる。これは装飾ではなく、Google画像検索、Pinterest、無断転載先で画像が単体流通したときのブランディング対策です。

- 背景が薄い場合: 透過ロゴを使う
- 背景が濃い・複雑な場合: 薄い白背景付きロゴを使う
- 配置: 上部または下部の余白から、人物の顔、重要な品物、図解ラベルを邪魔しない場所を選ぶ
- 画像生成AIにはロゴを描かせない

## 画風とトーン

- 温かみのある絵本風・水彩寄り・編集イラスト調
- 日本の現代的な実家、和室、リビング、古い家具、段ボール、古い品物
- やさしい、清潔、少し生活感がある
- 不安を煽らない
- 50代以上にも親しみやすい
- 買取業者っぽくしすぎない
- 若者向けにポップすぎない

親子を描写する場合:

- 親は70〜80代
- 子世代は40〜60代
- 若い親子や幼い子供にしない

## NG表現

- 真っ白で生活感がないミニマリスト部屋
- 高級感だけが強い買取業者風の画像
- 汚部屋、ゴミ屋敷感が強すぎる画像
- 暗い遺品整理、孤独、恐怖を煽る画像
- 現金や高額査定を強調する画像
- 画像生成AIが描いたロゴ、透かし、看板、不要文字
- 図解の文字詰め込みすぎ
- 抽象図形だけで構成された低品質図解

## 自動QAと再生成

Codex側では、生成後に以下を確認する。

- サイズと比率が合っている
- 不要な文字、ロゴ、透かしが混ざっていない
- 暗すぎない、汚すぎない、怖すぎない
- 記事文脈に合う人物・品物・場所が入っている
- 図解が抽象図形だけになっていない
- ロゴが入っている
- ロゴが顔、重要な品物、図解ラベルを邪魔していない
- ファイル名、ALT、キャプション、設置位置が画像プランと一致している

不合格の場合は最大2回まで自動再生成し、それでも通らない場合は要確認として人間判断に戻す。

## 最終形式と納品

- 標準形式はPNG
- 容量が大きい写真風画像だけJPEGまたはWebP候補
- WordPress側にWebP化プラグインがあるため、無理な過圧縮はしない
- Codexは `assets/images/<slug>/` に完成画像を置き、READMEに以下をまとめる

READMEに残す項目:

- 用途
- ファイル名
- サイズ
- WordPress画像タイトル
- ALT
- キャプション候補
- 設置位置
- ロゴ表示モード
- 要確認があればその理由

## 完了条件

- 記事内に5枚分のCMSブリーフがある
- `image-plan.json` と `image-plan.md` が生成されている
- 記事内にALT、キャプション候補、設置位置などのWordPress作業用CMSブリーフがある
- 記事内に英語の完成プロンプトやDALL-E指定が埋め込まれていない
- アイキャッチ素材はCanva手動仕上げ前提になっている
- 本文画像・図解4枚はブランド帰属ロゴ必須になっている
- 図解は抽象図形ではなく、サンプルのようなイラスト図解として作る前提になっている
