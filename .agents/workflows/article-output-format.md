# 記事出力フォーマットルール

記事を `.md` ファイルとして出力する際、以下のフォーマットに従うこと。
目的は「WordPressへのコピペ入稿時に、不要な情報を消す手間をゼロにする」こと。

## ⛔ 最重要ルール

1. **1ファイル＝1記事**。設計用ファイルと入稿用ファイルを分けない。
2. **本文中に「PHASE」「H2-1」等の制作用ラベルを絶対に書かない。**
3. ファイル冒頭のHTMLコメント（凍結済み設計図）は、WordPress上では非表示になるため、消す必要がない。
4. 本文はHTMLコメントの直後から始め、**そのままコピペで入稿できる状態**にすること。

## ファイル構成

```
<!-- === 凍結済み設計図（Invariants） === ... === END === -->

# タイトル（H1）

導入文

## H2見出し（番号なし）

本文...

## よくある質問

Q. 質問文
A. 回答文

## まとめ

本文...
```

## 凍結済み設計図のフォーマット

HTMLコメント `<!-- === 凍結済み設計図（Invariants） === ... === END === -->` の中に、以下の【】セクションを含めること。

### 必須セクション

```
【記事基本情報】
- メインKW: 
- スラッグ: 
- Meta Description: 
- 記事タイプ: 

【採用タイトル】
（確定したタイトル1本）

【ターゲット読者】
（誰の・どんな悩みを解決するか）

【検索意図】
（読者が何を知りたくて検索しているか）

【Out of Scope（扱わないこと）】
- （この記事では扱わないトピック）

【見出し構成案とインサイト】
H2: 見出しテキスト
  H3: 見出しテキスト
  H3: 見出しテキスト
H2: 見出しテキスト
...

【まとめ・CTA設計案】
まとめ：（要点の方向性）
CTA：（誘導先がある場合）
```

### 任意セクション（必要に応じて追加）
- 【内部リンク設計】
- 【競合との差別化ポイント】
- 【画像設計】

## 本文のルール

### 導入文（悩みへの共感＋KWとトピックの宣言＋Kackyからの呼びかけ）
- 導入文は、必ず以下の構成で記述すること。※固定の挨拶文（こんにちは。〜セイリ部です）は通常の記事には不要（トップページ・ピラー記事のみ）。
  1. **吹き出し（読者の悩み）**：ステレオタイプを避け、読者の喉元まで出ているリアルで生々しい痛みを代弁して「あ、私のことだ」と深い共感を呼ぶ。
  2. **概要の提示**：ターゲットKWを含め、「この記事では〜を解説します」とシンプルに宣言する（※この内容を後半で反復しないこと）。
  3. **リード文本文**：AI特有のツギハギ感を排除し、前の文から滑らかに接続する。「なぜそれが難しいのか」課題を炙り出して寄り添い、「この記事を読めばどう心が軽くなるのか（ベネフィット）」という強い動機づけを行う。
  4. **吹き出し（Kacky）**：冒頭の悩みをただオウム返しするのではなく、頼もしい案内人として読者の背中を押し、「〜方に参考にしていただければ嬉しいです！」等で結ぶ。
  5. **この記事でわかること**：4つの箇条書き。
- 以下のフォーマットに厳密に従うこと。

```markdown
🗨️吹き出し（読者の悩み）：
（「親は『まだ使う』と怒るし、一体どこから手をつければ…」等、生々しい読者の声）

この記事では、そんな【ターゲットKW】の最適な処分方法や〜〜について解説します。

（リード文本文：概要から滑らかに接続。問題の炙り出しと深い共感、そしてこの記事を読むとどう解決するのかのベネフィット提示）

🗨️吹き出し（Kacky）：
（頼もしい案内人として背中を押し、「〜方に参考にしていただければ嬉しいです！」等で結ぶ）

**この記事でわかること**
- 箇条書き1
- 箇条書き2
- 箇条書き3
- 箇条書き4
```

### 見出し
- `##` はH2、`###` はH3で出力。番号（①②、H2-1等）は付けない。
- 読者が「何が書いてあるか」をひと目で判断できる見出しにする。

### 内部リンク
- `> **あわせて読みたい：**` の引用ブロック形式で、説明1文＋リンクを配置。

### 画像制作ブリーフ
- **1記事に必要な画像一式**: アイキャッチ1枚 ＋ 記事の内容に応じた十分な数の本文図解（目安3〜5枚） ＋ 必要に応じて情景イラスト。
- **図解の出し惜しみ厳禁**: 読者の理解を助ける箇所（手順、比較、チェックリスト、アフィリエイト導線など）には、積極的に図解を配置すること。
- 記事エージェントは「目的・読者に伝えたい感情・入れたい要素・避けたい表現」までを指定する。画像の種別（編集イラスト/図解/写真風素材等）は画像制作側が判断する。
- セイリ部のブランドでは、写真よりも**温かい編集イラスト**が基本。写真風素材は「品物の実物感」が重要な記事に限定する。

#### アイキャッチ画像ブリーフ（必須）
すべての記事に必ず1枚、以下のフォーマットでアイキャッチの制作ブリーフを記述すること。設置位置は記事ファイルの末尾（まとめセクションの後）。
```
> [!CMS担当者へ：アイキャッチ画像]
> - 目的：
> - 読者に伝えたい感情：
> - 構図：
> - 入れたい人物・物：
> - 避けたい表現：
> - タイトル文言：
> - サブタイトル文言：
> - ALT：
> - ファイル名：
```

#### 本文図解ブリーフ（本文中の該当箇所に配置）
読者の理解を助ける図解やインフォグラフィックを、本文中の該当箇所に引用ブロック形式で配置する。「完成プロンプト」ではなく、画像制作者に意図を伝える「制作ブリーフ」として記述すること。
```
> [!CMS担当者へ：ここに図解を挿入]
> - 目的：
> - 読者に伝えたい感情：
> - 入れたい要素：
> - 避けたい表現：
> - ALT：
> - キャプション候補：
```

### FAQ
- `Q. **質問文**` と `A. 回答文` のペアで書く。
- 質問文は太字にする。
- 構造化データ（schema.org FAQPage）に変換可能な形式を維持。

### 禁止事項
- 本文中に `---`（水平線）を使わない。
- 本文末尾にメタ情報を別途書き出さない（凍結済み設計図に集約する）。
- frontmatter（YAMLの`---`ブロック）は入稿用ファイルには使わない。
