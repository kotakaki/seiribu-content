# 記事出力フォーマットルール

記事を `.md` ファイルとして出力する際、以下のフォーマットに従うこと。
目的は「WordPressへのコピペ入稿時に、不要な情報を消す手間をゼロにする」こと。

## ⛔ 最重要ルール

1. **1ファイル＝1記事**。設計用ファイルと入稿用ファイルを分けない。
2. **本文中に「PHASE」「H2-1」等の制作用ラベルを絶対に書かない。**
3. ファイル冒頭のHTMLコメント（凍結済み設計図）は、WordPress上では非表示になるため、消す必要がない。
4. 本文はHTMLコメントの直後から始め、**そのままコピペで入稿できる状態**にすること。

## ファイル構成

```
<!-- === 凍結済み設計図（Invariants） === ... === END === -->

# タイトル（H1）

導入文

## H2見出し（番号なし）

本文...

## よくある質問

Q. 質問文
A. 回答文

## まとめ

本文...
```

## 凍結済み設計図のフォーマット

HTMLコメント `<!-- === 凍結済み設計図（Invariants） === ... === END === -->` の中に、以下の【】セクションを含めること。

### 必須セクション

```
【記事基本情報】
- メインKW: 
- スラッグ: 
- Meta Description: 
- 記事タイプ: 

【採用タイトル】
（確定したタイトル1本）

【ターゲット読者】
（誰の・どんな悩みを解決するか）

【検索意図】
（読者が何を知りたくて検索しているか）

【Out of Scope（扱わないこと）】
- （この記事では扱わないトピック）

【見出し構成案とインサイト】
H2: 見出しテキスト
  H3: 見出しテキスト
  H3: 見出しテキスト
H2: 見出しテキスト
...

【FAQ設計案】
Q: 
A: 

【まとめ・CTA設計案】
まとめ：（要点の方向性）
CTA：（誘導先がある場合）
```

### 任意セクション（必要に応じて追加）
- 【内部リンク設計】
- 【競合との差別化ポイント】
- 【画像設計】

## 本文のルール

### 導入文（コピーライティングの基本に基づく自然な流れ）
- 導入文は、読者の感情に寄り添い、以下の自然な流れ（PASONAの法則）で記述すること。※固定の挨拶文（こんにちは。〜セイリ部です）は通常の記事には不要（トップページ・ピラー記事のみ）。
  1. **背景提示（地の文）**：ターゲットKWを自然に含めつつ、読者が直面している状況を1〜2文で描写する。**※「〜はありませんか？」のような直接的な問いかけは読者の心理的防御を生むため厳禁。「〜ということがあります」「〜という方も多いかもしれません」といった客観的な事実（一般論）として語ること。**
  2. **読者の心の声（鉤括弧）**：「〇〇が怖い」「〇〇で不安…」など、喉元まで出ている生々しい痛みを鉤括弧（「」）を用いて代弁する。※「🗨️吹き出し」などの無機質なAI的ラベルは使用しない。
  3. **「あるある」の一般言語化と分析**：読者の心の声の直後に、その生々しい感情を「決して珍しいことではない」と一般現象として言語化する一文を挟み、共感を深める。続けて「実は、このような状況は〜という状態によく見られます」と**客観的な事実や分析**を提示する。これにより、読者が逆に「まさに自分のことだ」と自発的に自分事化できるようにする。
  4. **解決策の提示と宣言**：「しかし、〇〇を知っていれば防げます。この記事では〜〜について解説します」と、希望を持たせた上で記事のテーマを宣言する。
  5. **吹き出し（Kacky）**：頼もしい案内人として実務的なアドバイスで背中を押す。※過度に明るい煽りや根拠のない断定は禁止。
  6. **この記事でわかること**：4つの箇条書き。
- 以下のフォーマットに厳密に従うこと。

#### 通常記事の導入文テンプレート

```markdown
（背景提示：ターゲットKWを含め、「〜ということがあります」等の客観的な事実として描写。※「〜ありませんか？」は禁止）

「（読者の心の声1：生々しい悩みや不安）」
「（読者の心の声2）」

（あるあるの言語化：「このように〜と感じるのは、決して珍しいことではありません」等）
（客観的な事実・分析の提示：「実は、このようなケースは〜によく見られます」等、一般論として語ることで読者に自分事化させる）
（課題の深掘り・解決策の提示：しかし、ポイントを知っていれば解決できる等のベネフィット提示）

この記事では、（ターゲット読者）に向けて、（ターゲットKWを用いた記事の概要）を解説します。

🗨️吹き出し（Kacky）：
「（実務的なアドバイスや案内人としての力強い一言で背中を押す）」

**この記事でわかること**
- 箇条書き1
- 箇条書き2
- 箇条書き3
- 箇条書き4
```

#### ピラー記事（トップページ含む）の導入文テンプレート
※通常記事との違いは「挨拶文」が冒頭に入ること。導入文の構造自体は同じPASONAの流れを踏襲する。

```markdown
こんにちは。実家の片付けに悩む方に向けて、捨てる前にモノと気持ちを整理し、「売る・残す・処分する」の選択をサポートするブログ『セイリ部』です。

（背景提示：ターゲットKWを含め、「〜ということがあります」等の客観的な事実として描写。※「〜ありませんか？」は禁止）

「（読者の心の声1：生々しい悩みや不安）」
「（読者の心の声2）」

（あるあるの言語化）
（客観的な事実・分析の提示）
（課題の深掘り・解決策の提示）

この記事では、（ターゲット読者）に向けて、（ターゲットKWを用いた記事の概要）を解説します。

🗨️吹き出し（Kacky）：
「（実務的なアドバイスや案内人としての力強い一言で背中を押す）」

**この記事でわかること**
- 箇条書き1
- 箇条書き2
- 箇条書き3
- 箇条書き4
```

### 見出し
- `##` はH2、`###` はH3で出力。番号（①②、H2-1等）は付けない。
- 読者が「何が書いてあるか」をひと目で判断できる見出しにする。

### 太字（Bold）のルール
- **括弧と太字の位置関係**: 鉤括弧「」や丸括弧（）を太字にする場合、`**` は必ず括弧の**内側**に置く。
  - 正：「**テキスト**」
  - 誤：**「テキスト」**
- **強調は最小限に**: 本当に重要な箇所のみ太字にし、乱用しない。
- **自然な文脈に溶け込ませる**: 不自然な鉤括弧囲みは避け、文章の流れの中で太字を使う。

### 内部リンク
- `> **あわせて読みたい：**` の引用ブロック形式で、説明1文＋リンクを配置。

### 本文画像ブリーフ
- **1記事に必要な画像一式**: アイキャッチ1枚 ＋ 記事の内容に応じた十分な数の画像（目安3〜5枚）。
- **図解と情景イラストのバランス**: B2B向けのコンサル資料ではないため、比較表やフローチャートなどの「図解」ばかりに偏らないこと。読者の感情に寄り添い、雰囲気を伝える「情景イラスト」や「写真風素材」のアイデア（例：実家の段ボールの山、笑顔で話し合う親子など）をバランス良く盛り込むこと。
- 記事エージェントは「種類・目的・読者に伝えたい感情・入れたい要素（アイデア）・避けたい表現」までを指定する。完成プロンプト、文字入れ、ロゴ配置、フォント指定は画像制作側またはローカル合成工程に渡す。完成版のセイリ部ロゴは画像制作側で必須配置するため、記事エージェントが個別に指示しなくてよい。
- セイリ部のブランドでは、写真よりも**温かい編集イラスト**が基本。写真風素材は「品物の実物感」が重要な記事に限定する。

#### アイキャッチ画像ブリーフ（必須）
すべての記事に必ず1枚、以下のフォーマットでアイキャッチの制作ブリーフを記述すること。設置位置は記事ファイルの末尾（まとめセクションの後）。
※画像制作エンジンは「正方形のイラスト素材のみ」を生成するため、文字や背景を含まないよう明確に指示すること。
```
> [!CMS担当者へ：アイキャッチ画像]
> - 種類：アイキャッチ素材（文字や背景を含まない、人物と小物のイラスト素材のみ）
> - 目的：
> - 読者に伝えたい感情：
> - 構図：
> - 入れたい人物・物：
> - 避けたい表現：画像内へのテキスト（文字）の生成、背景の描画、（その他記事特有のNG表現）
> - タイトル文言：（※画像内には生成せず合成時に追加）
> - サブタイトル文言：（※同上）
> - ALT：
> - ファイル名：
```

#### 本文画像ブリーフ（本文中の該当箇所に配置）
読者の理解を助ける図解や、感情に寄り添う情景イラストを、本文中の該当箇所に引用ブロック形式で配置する。「完成プロンプト」ではなく、画像制作者に意図を伝える「制作ブリーフ」として記述すること。
```
> [!CMS担当者へ：ここに画像を挿入]
> - 種類：記事内イメージ素材 / 写真風素材 / 記事内図解素材 / 図解用小物素材
> - 画像の目的：
> - 読者に伝えたい感情：
> - 画像のアイデア（情景・図解の具体案）：
> - 避けたい表現：画像内へのテキスト（文字）の生成、（その他記事特有のNG表現）
> - ALT：
> - キャプション候補：
```

### FAQ
- `Q. ` と `A. ` で始めるペアで書く（QやAの後に半角スペースを入れること）。
- **質問文のテキストのみを太字にする**（`Q.` 自体は太字に含めないこと）。
  - 正：`Q. **質問文**`
  - 誤：`**Q. 質問文**`
- 構造化データ（schema.org FAQPage）に変換可能な形式を維持。

### 禁止事項
- 本文中に `---`（水平線）を使わない。
- 本文末尾にメタ情報を別途書き出さない（凍結済み設計図に集約する）。
- frontmatter（YAMLの`---`ブロック）は入稿用ファイルには使わない。
